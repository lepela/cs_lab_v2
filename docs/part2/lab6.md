## 직접 체험 — 커넥터로 바꾸면 승인이 풀린다

읽기는 커넥터로 됐습니다. **쓰기(상태 변경)** 도 그냥 될까요? 직접 만져 봅니다.

1. SharePoint **본인 지원자 목록**에서 **승인된** 지원자 한 명을 골라, **전형단계**를 `서류합격`으로 직접 편집하고 저장합니다.

    ![촬영: SP 목록에서 전형단계 직접 편집](https://placehold.co/736x490/EEE/31343C "lab6-01")

2. 그 지원자의 **승인상태**를 확인합니다 — **보류 중으로 리셋**되어 있습니다.

    ![촬영: 승인상태가 보류 중으로 리셋](https://placehold.co/736x490/EEE/31343C "lab6-02")

3. 면접관 에이전트에 `지금 지원자 누구 있어?`를 물어봅니다. 방금 그 지원자가 **목록에서 사라졌습니다**(승인됨만 조회되므로).

    ![촬영: 그 지원자가 조회에서 사라짐](https://placehold.co/736x490/EEE/31343C "lab6-03")

    {: .important }
    전형단계 한 칸 바꿨을 뿐인데 승인이 풀려 사라졌습니다. 콘텐츠 승인 리스트는 **항목 수정 시 SharePoint가 승인상태를 리셋**합니다(서버 부작용) — 지침으로 못 막습니다. **대상 유일성·원자성·불변식**도 커넥터+지침으론 보장 안 됩니다. 그래서 상태 변경은 **흐름**이 맡습니다. 이제 그 흐름을 만듭니다.

---

## 흐름 빌드 — ID 단건 갱신 + 승인 보존

4. Copilot Studio **흐름**에서 **+ 새 에이전트 흐름**을 만들고 이름을 `전형단계 변경`으로 합니다.

    ![촬영: 전형단계 변경 흐름 만들기](https://placehold.co/736x490/EEE/31343C "lab6-04")

5. 트리거를 **에이전트가 흐름을 호출할 때**로 둡니다.

    ![촬영: 트리거 선택](https://placehold.co/736x490/EEE/31343C "lab6-05")

6. **+ 입력 추가** → 텍스트 **ID**. 설명: `대상 항목의 ID. 이름·상태 값을 넣지 말 것.`

    ![촬영: 입력 ID + 설명](https://placehold.co/736x490/EEE/31343C "lab6-06")

7. **+ 입력 추가** → 텍스트 **전형단계**. 설명: `목표 상태. 검토중/서류합격/면접합격/최종합격/불합격 중 하나.`

    ![촬영: 입력 전형단계 + 설명](https://placehold.co/736x490/EEE/31343C "lab6-07")

8. **+ 입력 추가** → 텍스트 **AI적합도**(선택적). 설명: `보조 입력. 불분명하면 빈 값.`

    ![촬영: 입력 AI적합도(선택적)](https://placehold.co/736x490/EEE/31343C "lab6-08")

    {: .warning }
    AI적합도를 **필수**로 두면 에이전트가 값을 못 채울 때 흐름이 멈춥니다(stall). **선택적**으로 둬야 합니다.

9. **+ 동작 추가** → SharePoint **항목 업데이트**를 추가하고 본인 계정으로 연결합니다.

    ![촬영: 항목 업데이트 추가](https://placehold.co/736x490/EEE/31343C "lab6-09")

10. **사이트 주소** = `SPSiteUrl` 칩, **목록 이름** = 본인 목록을 지정합니다.

    ![촬영: 사이트·목록 지정](https://placehold.co/736x490/EEE/31343C "lab6-10")

11. **ID** 칸에 `/`로 입력 **ID** 칩을 넣습니다.

    ![촬영: ID = 입력 ID](https://placehold.co/736x490/EEE/31343C "lab6-11")

12. **전형단계** 칸에 `/`로 입력 **전형단계** 칩을 넣습니다.

    ![촬영: 전형단계 매핑](https://placehold.co/736x490/EEE/31343C "lab6-12")

13. **AI적합도** 칸에 `/`로 입력 **AI적합도** 칩을 넣습니다. (종속 컬럼을 한 묶음으로 = 원자 갱신)

    ![촬영: AI적합도 매핑](https://placehold.co/736x490/EEE/31343C "lab6-13")

14. **+ 동작 추가** → SharePoint **콘텐츠 승인 상태 설정**. ID = 입력 ID 칩, **작업** = `Approve`(승인됨).

    ![촬영: 콘텐츠 승인 상태 설정(보존)](https://placehold.co/736x490/EEE/31343C "lab6-14")

    {: .important }
    **이게 흐름의 존재 이유입니다.** 업데이트가 1번에서 본 그 리셋을 일으키지만, 흐름이 **곧바로 다시 승인됨으로 설정**해 보존합니다. 커넥터 단독으론 불가능한 일을 흐름이 한 묶음으로 처리합니다.

15. **에이전트에 응답** 노드를 추가해 결과를 돌려줍니다.

    ![촬영: 응답 노드](https://placehold.co/736x490/EEE/31343C "lab6-15")

16. **저장 + 게시**합니다.

    ![촬영: 저장·게시](https://placehold.co/736x490/EEE/31343C "lab6-16")

17. 면접관 에이전트 **도구**에 이 흐름을 추가하고, 설명에 `지원자의 전형단계를 변경할 때 호출. ID로 단건 갱신하고 승인을 보존한다.`를 적습니다.

    ![촬영: 흐름을 도구로 추가](https://placehold.co/736x490/EEE/31343C "lab6-17")

---

## 확인

- [ ] SP 직접 편집 시 **승인 리셋**을 목격했다
- [ ] 흐름이 **ID 단건 갱신 + 콘텐츠 승인 상태 설정(보존)** 으로 구성됐다
- [ ] AI적합도 입력이 **선택적**이다(stall 방지)
- [ ] 흐름이 에이전트 **도구**로 붙었다

{: .important }
흐름은 **ID로 단건** 갱신만 합니다 — "누구인지"(이름→ID, 동명이인)와 "정말 할까요?"(사람 확인)는 흐름이 아니라 **토픽**이 맡습니다. 다음 **Lab 7**에서 그 게이트와 확인 카드를 얹어 면접 확정을 완성합니다.
