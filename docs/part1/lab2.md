이 랩 예상 **10분** (오전 통합 50분 중).

---

## 준비

강사가 제공한 **Knowledge 4종 PDF**를 내려받아 둡니다 — 채용 가이드 / 직무기술서(JD) / 회사 지침 / 평가 루브릭. Copilot Studio에 접속하고 우측 상단 환경이 **교육용 환경**인지 확인합니다(Lab 1에서 `HR 채용 자동화` 솔루션이 들어와 있는 그 환경).

---

## 단계

1. 왼쪽 메뉴 **에이전트** → **새 에이전트**를 클릭하고, 대화형 생성 화면에서 **구성으로 건너뛰기**를 누릅니다. (빈 에이전트로 시작해 구성 요소를 직접 다룹니다.)

    ![촬영: 새 에이전트 → 구성으로 건너뛰기](https://placehold.co/736x490/EEE/31343C "lab2-01")

2. **이름**에 `면접관 에이전트`를 입력합니다.

    ![촬영: 이름 입력](https://placehold.co/736x490/EEE/31343C "lab2-02")

3. **설명**에 `승인된 지원자를 조회·평가하고 면접 질문을 돕는 채용 도우미`를 입력합니다.

    ![촬영: 설명 입력](https://placehold.co/736x490/EEE/31343C "lab2-03")

4. **솔루션** 항목에서 Lab 1에서 들어온 **`HR 채용 자동화`** 솔루션을 선택합니다.

    ![촬영: 솔루션 = HR 채용 자동화 선택](https://placehold.co/736x490/EEE/31343C "lab2-04")

    {: .important }
    에이전트를 **이 솔루션에 넣어야** Lab 3에서 도구의 사이트 주소를 **환경 변수(SPSiteUrl)로 바인딩**할 수 있습니다. 다른 솔루션(기본 솔루션)에 있으면 입력값 선택의 **환경** 탭에 변수가 안 보여 바인딩이 안 됩니다.

5. **만들기**를 클릭합니다. 생성 후 우측 상단 **환경**이 교육용 환경이 맞는지 확인합니다.

    ![촬영: 에이전트 개요 + 환경 확인](https://placehold.co/736x490/EEE/31343C "lab2-05")

6. **설정 → 생성형 AI → 지식**에서 **근거 없는 응답 허용**이 **켜기(ON)** 인지 확인합니다(기본값 유지). 같은 화면의 **사용자 피드백 수집**은 끄기(기본값) 유지합니다.

    ![촬영: 근거 없는 응답 허용 ON 확인](https://placehold.co/736x490/EEE/31343C "lab2-06")

    {: .note }
    이 토글을 켜 둬야 도구가 없는 지금도 에이전트가 **"그건 실시간 데이터라 도구 없이 확인 못 한다"고 스스로 경계를 설명**합니다(아래 시연). 끄면 그런 응답이 폴백으로 차단됩니다.

7. **Knowledge** 탭에서 **+ 추가 → 파일 업로드**를 선택합니다.

    ![촬영: Knowledge 추가 → 파일 업로드](https://placehold.co/736x490/EEE/31343C "lab2-07")

8. `채용 가이드 문서.pdf`를 업로드하고 **설명**에 `공통 평가기준과 경력 레벨 정의`를 입력한 뒤 **에이전트에 추가**합니다.

    ![촬영: 채용 가이드 업로드 + 설명](https://placehold.co/736x490/EEE/31343C "lab2-08")

9. 같은 방식으로 나머지 3종을 업로드합니다(각 설명 입력).

    - `직무기술서(JD).pdf` → `직군별 필수·우대 요건 목록`
    - `회사 지침 문서.pdf` → `인재상 4축 및 채용 원칙`
    - `평가 루브릭.pdf` → `지원자 적합도를 등급으로 매기는 채점 규칙`

    ![촬영: 4종 모두 업로드된 Knowledge 목록](https://placehold.co/736x490/EEE/31343C "lab2-09")

    {: .note }
    **이력서 원문은 Knowledge에 넣지 않습니다.** 수백 개를 RAG에 올리면 검색이 코퍼스 전체에서 일어나 **동명이인·교차 오염**이 생깁니다. 이력서는 대상 데이터라 목록에 두고, 원문은 필요할 때 1건만 링크로 가져옵니다(Lab 3). 평가의 일상 재료는 적재 흐름이 만든 **이력서요약**입니다 — "요약=기본 / 원문=필요할 때만".

10. 4종 상태가 모두 **Ready**가 될 때까지 기다립니다. 대기 중 아래 지침 작성을 진행하고 돌아와 확인합니다.

    ![촬영: Knowledge 4종 Ready](https://placehold.co/736x490/EEE/31343C "lab2-10")

11. **지침** 영역에 아래를 그대로 붙여넣습니다. (지침은 **Markdown 문법**으로 작성합니다 — 짧고 구조적으로.)

    ```
    ## 역할
    누리커머스(주) 채용을 돕는 면접관 에이전트. 채용 담당자가 지원자를 조회·평가하고 면접을 준비하도록 돕는다. 한국어로 간결·정중하게, 기준(채용 가이드·JD·회사 지침·평가 루브릭)에 근거해 답하고 없는 내용은 단정하지 않는다.

    ## 자료
    - **Knowledge**(채용 가이드·JD·회사 지침·평가 루브릭) = 평가 기준. 채용 기준·적합도·인재상 질문은 여기서 답한다.
    - **지원자 목록**(SharePoint) = 대상 데이터. 특정 지원자·상태 질문은 실시간 조회가 필요하다.
    - **이력서 원문** = 증거. 특정 1명이 필요할 때만 링크로 연다.

    ## 표현
    내부 도구·시스템 식별자(Qv2 등)는 노출하지 않는다.
    ```

    ![촬영: 지침 3블록 붙여넣기](https://placehold.co/736x490/EEE/31343C "lab2-11")

    {: .note }
    지침은 한 번 쓰고 끝이 아닙니다. 이후 **테스트 → 실패 관찰 → 블록 추가 → 재테스트**로 유닛마다 누적됩니다. 지금은 기초 3블록만.

12. **저장**하고 **게시(Publish)**합니다.

    ![촬영: 지침 저장 + 게시](https://placehold.co/736x490/EEE/31343C "lab2-12")

    {: .warning }
    지침을 바꿔도 동작이 그대로면 **게시 여부**를 먼저 확인하세요. 저장만 하고 게시 안 해 옛 동작이 유지되는 경우가 흔합니다.

13. (시연) 10단계 지식이 모두 **Ready**인지 확인하고, 테스트 패널에 `백엔드 채용 기준이 뭐야?`를 입력합니다 → JD 근거 답변.

    ![촬영: 채용 기준 답변(JD 근거)](https://placehold.co/736x490/EEE/31343C "lab2-13")

14. `지원자 적합도는 어떻게 평가해?` → 평가 루브릭 근거 답변.

    ![촬영: 적합도 평가 기준 답변](https://placehold.co/736x490/EEE/31343C "lab2-14")

15. `우리 회사 인재상이 뭐야?` → 회사 지침 근거 답변.

    ![촬영: 인재상 답변](https://placehold.co/736x490/EEE/31343C "lab2-15")

16. `이력서에서 무엇을 추출해야 해?` → 평가 루브릭의 추출 스키마 근거 답변.

    ![촬영: 이력서 추출 항목 답변](https://placehold.co/736x490/EEE/31343C "lab2-16")

    {: .note }
    "무엇을 추출하느냐"는 곧 적재 흐름이 만든 데이터의 스키마인데, 그 정의가 **루브릭 안에** 있습니다(평가 축 ↔ 추출 필드 1:1). 에이전트가 자기 적재 기준을 **메타적으로** 설명하는 셈입니다.

17. `김지훈 지원했어? 지금 상태가 어때?` 와 `승인된 지원자 목록 보여줘` 를 입력합니다. 에이전트가 **답하지 못하고**, 지침 덕분에 "실시간 데이터라 도구가 없어 확인 불가"라고 **경계를 스스로 설명**하는지 확인합니다.

    ![촬영: 특정 지원자/목록 — 경계 설명 응답](https://placehold.co/736x490/EEE/31343C "lab2-17")

---

## 확인

- [ ] 면접관 에이전트가 **`HR 채용 자동화` 솔루션**에 생성됐다
- [ ] **근거 없는 응답 허용 = ON** 확인
- [ ] 지식 4종이 모두 **Ready**, 지침 3블록 저장·게시됨
- [ ] 기준 질문 4종에 Knowledge로 답한다
- [ ] 특정 지원자/목록은 **답 못 하고 경계를 스스로 설명**한다

{: .important }
에이전트는 **기준**(Knowledge)은 알지만 **지금 누가 지원했는지**(데이터)는 모릅니다. 이 간극을 메우는 **SharePoint 조회 도구**를 다음 Lab 3에서 붙입니다.
